# Design Document

## Why This Project

Energy profiling tools require hardware or runtime execution. This LLVM pass
provides a static estimate of relative energy consumption at the basic-block
level, enabling developers to identify energy hotspots without running code
or using power measurement tools.

## Architecture

The pass operates on LLVM IR with debug information:

```
Source → LLVM IR (-g -O0) → EnergyPass → Terminal/JSON/HTML Reports
                       ↓              ↓
               Block Frequency   Opcode Cost Model
               Info (BFI)        (JSON file)
```

### Pass Operation

1. **IR Processing**: For each function, get block frequencies from BFI
2. **Energy Calculation**: For each instruction:
   - Look up opcode cost in JSON model
   - Multiply by block's normalized frequency
   - Accumulate per-block, per-opcode, and per-source-line totals
3. **Reporting**: Output terminal summary, JSON report, and HTML heatmap

Key data structures:
- BlockReport: {name, frequency, energy}
- FuncReport: {name, total_energy, blocks[], hotspots[], advisories[], source_lines[]}

## Design Decisions

**Static Analysis**: No runtime needed, same input yields same output.
Trade-off: BFI frequencies are heuristic, energy is relative (unitless).

**JSON Energy Model**: Opcode costs in external file allow retargeting without
recompilation. Missing opcodes default to cost 1.0.

**Block Frequency Weighting**: Essential for capturing loop impact.
A block with 1000× frequency contributes 1000× the per-instruction cost.

**Three-Level Granularity**: Block (hotspots), opcode (advisors), source line
(heatmap) covers typical developer questions: "what function/block/line is hot?"

**Separate Visualization**: HTML report generated by Python script keeps
LLVM pass focused on analysis.

## Alternative Approaches Considered

- **MachineInstr Pass**: More accurate but backend-specific and less portable
- **Runtime Instrumentation (perf/RAPL)**: Accurate but requires execution and specific hardware
- **Hardcoded Cost Table**: Simpler but less flexible for architecture changes
